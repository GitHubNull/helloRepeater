## 界面概览

安装插件后，你可以在 Burp Suite 的 Tab 栏中看到 **helloRepeater** 标签页。

### 主界面结构

```
┌─────────────────────────────────────────────────────────────┐
│  helloRepeater │ Proxy │ Repeater │ ...                      │
├─────────────────────────────────────────────────────────────┤
│  ┌──────────────┬──────────────┬──────────────┐              │
│  │ 重命名规则   │  分组管理    │   规则测试   │              │
│  └──────────────┴──────────────┴──────────────┘              │
│                                                             │
│  [内容区域]                                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 三个核心标签页

| 标签页 | 功能描述 |
|-------|---------|
| **重命名规则** | 管理自动重命名规则，支持增删改查和搜索过滤 |
| **分组管理** | 创建和管理分组，支持层级结构和搜索 |
| **规则测试** | 测试规则匹配效果，调试规则配置 |

## 基础使用

### 发送到 Repeater Manager

这是插件的核心功能，支持三种发送方式：

#### 方式 1：直接发送（推荐日常使用）

**适用场景**：快速发送请求，使用默认规则自动命名

**操作步骤**：
1. 在 `Proxy` → `HTTP history` 中找到目标请求
2. **右键点击**请求
3. 选择 `发送到 Repeater Manager` → `直接发送`
4. 请求会自动发送到 Repeater，标签页名称根据规则自动生成

**示例效果**：
- 原始标签名：`1`
- 重命名后：`1-GET-/api/v1/users`

#### 方式 2：选择分组发送

**适用场景**：将请求发送到特定分组，便于分类管理

**操作步骤**：
1. 右键点击请求
2. 选择 `发送到 Repeater Manager` → `选择分组`
3. 在子菜单中选择目标分组（如 `API测试`、`漏洞验证`）
4. 请求会发送到 Repeater，标签页名称带有分组前缀

**示例效果**：
- 分组：`API测试`
- 标签名：`API测试/1-GET-/users/list`

#### 方式 3：新建分组并发送

**适用场景**：临时创建新分组并发送请求

**操作步骤**：
1. 右键点击请求
2. 选择 `发送到 Repeater Manager` → `新建分组...`
3. 在弹出的对话框中输入分组名称
4. 点击确定，请求会发送到新分组

### 在哪些界面可以使用右键菜单

右键菜单支持在以下 Burp Suite 工具中使用：

- ✅ **Proxy** → HTTP history / WebSockets history
- ✅ **Repeater** → 请求列表
- ✅ **Target** → Site map
- ✅ **Scanner** → 结果列表
- ✅ **Intruder** → 攻击结果
- ✅ **Extensions** → 其他插件的输出

## 重命名规则管理

### 规则列表界面

```
┌─────────────────────────────────────────────────────────────┐
│ 搜索过滤 [关键词] [模式▼] [忽略大小写 ☑] [搜索] [清除]        │
├─────────────────────────────────────────────────────────────┤
│ [添加规则] [编辑规则] [删除规则] | [导入规则] [导出规则]     │
├─────────────────────────────────────────────────────────────┤
│ 启用 │ 优先级 │ 名称        │ 类型          │ 模式          │
├──────┼────────┼─────────────┼───────────────┼───────────────┤
│  ☑   │   1    │ 路径后16字符 │ PATH_SUBSTRING│ last16        │
│  ☑   │   2    │ API版本提取  │ PATH_REGEX    │ /api/(v\d+)   │
│  ☐   │   3    │ JSON Action  │ BODY_JSON_PATH│ $.action      │
└─────────────────────────────────────────────────────────────┘
```

### 添加规则

**步骤**：
1. 在"重命名规则"标签页点击 `添加规则` 按钮
2. 在弹出的对话框中配置：

| 字段 | 说明 | 示例 |
|-----|------|------|
| 规则名称 | 规则的显示名称 | "提取用户ID" |
| 规则类型 | 5种类型之一 | PATH_REGEX |
| 匹配模式 | 根据类型填写 | `/users/(\d+)` |
| 优先级 | 数字越小优先级越高 | 1 |
| 启用 | 是否生效 | ☑ |
| 描述 | 规则说明 | "从URL提取用户ID" |

3. 点击 `保存`

### 编辑规则

**方法 1**：双击规则行
**方法 2**：选中规则后点击 `编辑规则`
**方法 3**：右键规则行选择 `编辑`

### 删除规则

1. 选中要删除的规则
2. 点击 `删除规则` 或右键选择 `删除`
3. 在确认对话框中点击 `是`

### 启用/禁用规则

- 点击规则行第一列的复选框即可切换启用状态
- 禁用的规则在重命名时会被跳过

### 搜索过滤功能

**搜索面板说明**：

```
关键词: [___________]  模式: [简单匹配 ▼]  [忽略大小写 ☑]  [搜索] [清除]
```

**搜索模式**：

| 模式 | 说明 | 示例 |
|-----|------|------|
| 简单匹配 | 关键词出现在任意字段 | 输入 "api" 匹配所有含 api 的规则 |
| 正则匹配 | 使用正则表达式匹配 | 输入 `v\d+` 匹配版本号规则 |

**搜索范围**：
- 规则名称
- 规则类型
- 匹配模式
- 描述

### 排序规则

规则默认按优先级排序，你可以：
1. **右键菜单** → `上移` / `下移` 调整优先级
2. 直接编辑规则的优先级数值

优先级说明：
- 数字越小，优先级越高
- 重命名时会按优先级顺序尝试规则
- 第一个匹配成功的规则会被使用

### 右键菜单功能

在规则表格上右键，可访问：

- **编辑** - 修改规则配置
- **删除** - 删除选中规则
- **上移** - 提高优先级（减 1）
- **下移** - 降低优先级（加 1）

## 分组管理

### 分组列表界面

```
┌─────────────────────────────────────────────────────────────┐
│ 搜索过滤 [关键词] [模式▼] [忽略大小写 ☑] [搜索] [清除]        │
├─────────────────────────────────────────────────────────────┤
│ [添加分组] [编辑分组] [删除分组] | [导入分组] [导出分组]     │
├─────────────────────────────────────────────────────────────┤
│ 名称        │ 父分组  │ 描述           │ 标签页数量         │
├─────────────┼─────────┼────────────────┼───────────────────┤
│ API测试     │ -       │ API接口测试     │ 5                 │
│ ├ 用户模块  │ API测试 │ 用户相关接口    │ 3                 │
│ ├ 订单模块  │ API测试 │ 订单相关接口    │ 2                 │
│ 漏洞验证    │ -       │ 验证的漏洞      │ 8                 │
└─────────────────────────────────────────────────────────────┘
```

### 添加分组

**步骤**：
1. 在"分组管理"标签页点击 `添加分组` 按钮
2. 配置分组信息：

| 字段 | 说明 | 示例 |
|-----|------|------|
| 分组名称 | 分组标识 | "API测试" |
| 父分组 | 上级分组（可选） | 选择已有分组或"无" |
| 描述 | 分组说明 | "用于API接口测试" |

3. 点击 `保存`

**层级分组示例**：
```
API测试
├── 用户模块
├── 订单模块
└── 支付模块
漏洞验证
├── SQL注入
├── XSS
└── 越权
```

### 编辑分组

- 双击分组行
- 或选中后点击 `编辑分组`
- 可修改名称、父分组和描述

**注意**：修改父分组时系统会自动检查循环引用，防止出现 A→B→A 的情况。

### 删除分组

1. 选中要删除的分组
2. 点击 `删除分组`
3. 确认删除

**注意**：删除分组不会影响已发送的 Repeater 标签页，只是移除分组记录。

### 搜索过滤分组

与规则搜索类似，支持：
- 简单匹配：按名称、描述搜索
- 正则匹配：使用正则表达式
- 忽略大小写选项

### 测试数据示例

**示例 1：简单的 GET 请求**
```http
GET /api/v2/products/12345 HTTP/1.1
Host: api.example.com
```

**示例 2：带 JSON Body 的 POST 请求**
```http
POST /api/users/create HTTP/1.1
Host: api.example.com
Content-Type: application/json

{
    "action": "createUser",
    "username": "testuser",
    "role": "admin"
}
```

**示例 3：带 XML Body 的请求**
```http
POST /soap/service HTTP/1.1
Host: api.example.com
Content-Type: text/xml

<?xml version="1.0"?>
<request>
    <method>getUserInfo</method>
    <id>12345</id>
</request>
```

## 导入导出

### 导出规则

**步骤**：
1. 在"重命名规则"标签页点击 `导出规则`
2. 选择保存位置
3. 文件将保存为 YAML 格式（如 `rules.yaml`）

**导出格式示例**：
```yaml
rules:
  - id: "rule-001"
    name: "路径后16字符"
    type: "PATH_SUBSTRING"
    pattern: "last16"
    priority: 1
    enabled: true
    description: "默认规则"
  - id: "rule-002"
    name: "API版本提取"
    type: "PATH_REGEX"
    pattern: "/api/(v\\d+)/.*"
    priority: 2
    enabled: true
    description: "提取API版本号"
```

### 导入规则

**步骤**：
1. 点击 `导入规则`
2. 选择之前导出的 YAML 文件
3. 确认导入
4. 系统会显示成功导入的规则数量

### 导出分组

与规则导出类似，在"分组管理"标签页操作。

**导出格式示例**：
```yaml
groups:
  - id: "group-001"
    name: "API测试"
    parentId: null
    description: "API接口测试分组"
  - id: "group-002"
    name: "用户模块"
    parentId: "group-001"
    description: "用户相关接口"
```

### 数据迁移

**场景 1：备份配置**
```
导出规则 → 保存 rules.yaml
导出分组 → 保存 groups.yaml
```

**场景 2：迁移到新环境**
```
旧环境：导出规则和分组
新环境：安装插件 → 导入规则和分组
```

**场景 3：团队共享**
```
一人配置：设置好规则和分组 → 导出
团队其他人：导入配置文件
```

## 实用技巧

### 技巧 1：批量管理标签页

虽然不能直接重命名已存在的 Repeater 标签页，但你可以：
1. 在 Proxy History 中选择多个请求（Ctrl+点击）
2. 右键批量发送到 Repeater Manager
3. 所有新标签都会应用规则自动命名

### 技巧 2：快速查找分组

当分组较多时：
1. 使用搜索功能快速定位
2. 输入关键词如 "API" 过滤所有 API 相关分组
3. 使用正则匹配更精确筛选

### 技巧 3：规则优先级调整

**高优先级规则**（优先级 1-10）：
- 特定项目的命名规则
- 紧急使用的规则

**中优先级规则**（优先级 11-50）：
- 通用的 API 命名规则
- 常用的正则提取规则

**低优先级规则**（优先级 51+）：
- 备用规则
- 默认兜底规则

### 技巧 4：组合使用规则和分组

**场景：API 测试**
```
1. 创建分组：API测试/用户模块
2. 配置规则：提取 /api/{version}/{resource}
3. 发送请求到该分组
4. 结果标签名：API测试/用户模块/1-GET-/api/v1/users
```

### 技巧 5：临时禁用规则

测试新规则时：
1. 暂时禁用旧规则（取消勾选）
2. 添加并启用新规则
3. 测试效果
4. 根据需要重新启用旧规则
